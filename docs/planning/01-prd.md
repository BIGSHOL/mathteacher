|---|------|------|

## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

학원에서 강사 한 명이 여러 학생을 가르치는 상황을 상상해보세요. 마치 **한 명의 요리사가 동시에 30개의 요리를 만들어야 하는 것**과 같습니다.

- 학생 A는 분수를 모르고
- 학생 B는 방정식에서 막히고
- 학생 C는 이미 다 이해했는데 지루해하고...

강사는 한 명씩 돌아다니며 "이해했어?", "여기 풀어봐"를 반복합니다. **학생 30명을 개별 확인하는 데 수업 시간의 절반이 소요**됩니다.

### 1.2 목표 상황 (To-Be)

**자동 주문 시스템이 있는 레스토랑**처럼, 학생들이 스스로 주문(테스트)하고, 즉시 결과를 받고, 부족한 부분을 알 수 있습니다.

- 학생: 혼자서도 테스트하고 즉시 결과 확인
- 강사: 대시보드에서 전체 학생 현황 한눈에 파악
- 결과: 강사는 "정말 도움이 필요한 학생"에게 집중 가능

### 1.3 왜 지금인가? (Why Now)

- **강사 채용 어려움**: 좋은 강사를 구하기 점점 어려워지는 상황
- **효율 극대화 필요**: 현재 강사 수를 유지하면서 더 많은 학생 관리 필요
- **자기주도 학습 트렌드**: 스스로 공부하는 능력이 점점 중요해짐

## 2. 사용자 페르소나

### 2.1 주요 페르소나 A: 학생

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "열심이 민수" (초등 5학년) |
| 나이/직업 | 12세 초등학생 |
| 핵심 니즈 | 재미있게 수학 공부하고 싶음, 칭찬받고 싶음 |
| 불편 포인트 | 문제집은 지루함, 틀려도 바로 답을 못 봄 |
| 사용 상황 | 학원 수업 전 예습, 집에서 숙제 |

### 2.2 주요 페르소나 B: 강사/관리자

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "바쁜 김쌤" |
| 나이/직업 | 28세 수학 강사 |
| 핵심 니즈 | 학생별 수준 빠르게 파악, 업무 시간 절감 |
| 불편 포인트 | 개별 확인에 시간이 너무 많이 듦 |
| 사용 상황 | 수업 전 학생 현황 확인, 주간 리포트 분석 |

### 2.3 페르소나의 하루 (시나리오)

**민수의 하루:**

아침 등원 전, 민수는 스마트폰으로 앱을 켭니다. "오늘의 미션: 분수 개념 테스트!"라는 메시지와 함께 귀여운 캐릭터가 인사합니다.

5문제를 풀자 "정답률 80%! 레벨업!" 팡파레가 울립니다. 틀린 1문제는 바로 해설을 보여주고, "이 부분 복습해볼까?"라고 친근하게 안내합니다.

학원에 도착하니, 김쌤이 "민수야, 분수 거의 다 이해했네! 오늘은 한 단계 더 해볼까?"라고 말합니다. 민수는 뿌듯합니다.

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (온보딩/로그인)

- [ ] 학생으로서, 선생님이 만들어준 계정으로 로그인할 수 있어야 한다. 왜냐하면 따로 가입 절차 없이 바로 학습을 시작할 수 있기 때문이다.
- [ ] 강사로서, 학생 계정을 쉽게 생성/관리할 수 있어야 한다. 왜냐하면 새 학생이 왔을 때 빠르게 시작할 수 있기 때문이다.
- [ ] 학생으로서, 로그인 상태가 유지되어야 한다. 왜냐하면 매번 로그인하면 귀찮기 때문이다.

### FEAT-1: 개념 테스트 (MVP)

- [ ] 학생으로서, 오늘의 개념 테스트를 시작할 수 있어야 한다. 왜냐하면 매일 조금씩 공부하면 실력이 늘기 때문이다.
- [ ] 학생으로서, 문제를 풀면 바로 정답/오답을 알 수 있어야 한다. 왜냐하면 기다리지 않고 즉시 확인해야 집중이 유지되기 때문이다.
- [ ] 학생으로서, 틀린 문제의 해설을 볼 수 있어야 한다. 왜냐하면 왜 틀렸는지 알아야 다음에 맞출 수 있기 때문이다.
- [ ] 학생으로서, 테스트 완료 후 "레벨업!", "콤보!" 같은 재미있는 피드백을 받아야 한다. 왜냐하면 그래야 계속 하고 싶어지기 때문이다.
- [ ] 강사로서, 학생별 개념 테스트 결과를 한눈에 볼 수 있어야 한다. 왜냐하면 누가 어디서 막혔는지 빠르게 파악해야 하기 때문이다.

### FEAT-2: 연산 연습 (v2 이후)

- [ ] 학생으로서, 사칙연산을 빠르게 반복 연습할 수 있어야 한다. 왜냐하면 연산 속도가 빨라야 어려운 문제도 풀 수 있기 때문이다.
- [ ] 학생으로서, 시간 제한 모드로 게임처럼 도전할 수 있어야 한다. 왜냐하면 긴장감이 있어야 집중력이 올라가기 때문이다.

### FEAT-3: 학습 통계/대시보드 (v2 이후)

- [ ] 강사로서, 반 전체의 취약 단원을 파악할 수 있어야 한다. 왜냐하면 수업 내용을 조절할 수 있기 때문이다.
- [ ] 학생으로서, 내 성장 그래프를 볼 수 있어야 한다. 왜냐하면 실력이 느는 걸 눈으로 확인하면 동기부여가 되기 때문이다.

## 4. 성공 지표

### 4.1 노스스타 지표 (결과 지표)

| 지표명 | 정의 | 목표치 | 측정 주기 |
|--------|------|--------|----------|

|--------|------|--------|

| 평균 정답률 향상 | 월간 평균 정답률 상승폭 | +10% |

## 5. 범위 관리

### 5.1 MVP (반드시 포함)

- **FEAT-1**: 개념 테스트 (출제, 풀이, 즉시 채점, 해설)
- **FEAT-0**: 학원 전용 계정 시스템 (강사가 학생 계정 관리)
- 듀오링고 스타일 게임화 피드백 (레벨업, 콤보, 스트릭)

### 5.2 옵션 (시간 허용 시)

- FEAT-2: 연산 연습 (타이머 모드)
- 간단한 강사 대시보드

### 5.3 Non-goals (명시적 제외)

- **AI 맞춤 추천**: 복잡한 추천 알고리즘은 v2 이후 (단순 레벨 분류만 MVP)
- **학부모 앱/알림**: 학원 내부 도구로 시작, 확장은 나중에
- **결제 시스템**: 학원 내부 전용이므로 불필요
- **오프라인 모드**: 클라우드 기반, 항상 온라인 가정

## 6. 가정과 리스크

### 6.1 핵심 가정

| # | 가정 | 검증 방법 |
|---|------|----------|
| 1 | 학생들이 게임화 요소에 동기부여된다 | 초기 사용자 5명 피드백 |
| 2 | 강사가 대시보드를 실제로 활용한다 | 주간 접속 로그 분석 |
| 3 | 학부모가 집에서 숙제 용도로 사용시킨다 | 가정 접속 비율 추적 |

### 6.2 Top 5 리스크 보드

| # | 가정 | 리스크 | 완화 | 트리거 | 대응 |
|---|------|--------|------|--------|------|
| 1 | 게임화가 동기부여 | 학생이 재미없어서 안 씀 | 듀오링고 스타일 벤치마크 | 주간 완료율 < 30% | 피드백 강화, 보상 추가 |
| 2 | 즉시 채점이 핵심 | 서버 지연으로 UX 저하 | 응답 < 500ms 목표 | P95 > 1초 | 캐싱, 인프라 확장 |
| 3 | 강사 활용 | 기존 방식 고수 | 간단한 대시보드 | 강사 접속 < 주 1회 | 교육, UX 개선 |
| 4 | 문제 품질 | 문제가 너무 쉽거나 어려움 | 난이도 분류 체계 | 정답률 > 95% or < 30% | 난이도 조정 |
| 5 | 반응형 지원 | 모바일에서 불편 | 모바일 퍼스트 설계 | 모바일 이탈률 높음 | UI 개선 |

## 7. 해결 가설과 실험

### 7.1 가설

> 만약 **수학 테스트에 듀오링고 스타일 게임화 요소(레벨업, 콤보, 스트릭)**를 적용하면, **학생들의 자발적 테스트 완료율이 70% 이상**이 될 것이다. 왜냐하면 **즉각적인 보상과 성취감이 학습 동기를 높이기 때문**이다.

### 7.2 실험 루프

```
실험 → 관측 → 학습 → 다음 가설
```

| 단계 | 내용 |
|------|------|
| 실험 | MVP 출시 후 2주간 학생 5명 테스트 |
| 관측 | 주간 테스트 완료율, 재방문율, 피드백 |
| 학습 | 70% 미달 시 실패 원인 분석 (재미? 난이도? UX?) |
| 다음 | 실패 원인별 개선 가설 수립 |
